# FamilyLedger — Project Context

Reference document for Claude sessions working on this codebase. Generated by reading
the actual source (not inferred from names) on 2026-07-20. The README describes this as
"Phase 1 — Architecture & Database Foundation... No UI screens are implemented yet," and
several doc comments in `lib/core/` repeat that claim — **this is stale**. The actual
`lib/features/` tree (~8,100 lines) contains fully wired screens, widgets, view models,
and providers for Dashboard, People, Reports, Transactions, Backup, and Settings, all
reachable from `AppShell`'s bottom navigation. Treat this document, not the README or old
doc comments, as the source of truth for what exists.

## 1. Architecture Overview

Layering as actually implemented:

```
UI (Widgets, ConsumerWidget/ConsumerStatefulWidget)
   ↓ ref.watch / ref.read
Riverpod Providers & ViewModels (StateNotifier / Notifier, per feature)
   ↓ calls interface methods
Repository Interfaces (lib/repositories/*.dart — abstract interface class)
   ↓ implemented by
Repository Implementations (lib/repositories/impl/*.dart)
   ↓ query directly against
Drift AppDatabase (generated table accessors: _database.select/into/update/delete)
   ↓ backed by
SQLite (via sqlite3_flutter_libs, NativeDatabase.createInBackground)
```

Key characteristics:

- **No separate DAO layer.** There is no `*Dao` class between repository impls and
  Drift. Each `*RepositoryImpl` (e.g. `TransactionRepositoryImpl`) queries the
  generated `AppDatabase` table accessors (`_database.select(_database.transactions)`,
  etc.) directly. The repository interface *is* the DAO boundary — this is a deviation
  from a textbook Clean Architecture where a DAO/data-source layer sits below the
  repository. Given Drift's generated table API is already a reasonably clean
  abstraction, this collapses two layers into one deliberately.
- **GetIt is the composition root, Riverpod is the consumption layer.** All concrete
  singletons (database, repositories, backup/export services) are registered once in
  `lib/core/services/service_locator.dart` via `GetIt`. `lib/providers/
  repository_providers.dart` then exposes each repository through a plain Riverpod
  `Provider<T>` that just does `sl<T>()`. Riverpod providers never construct
  repositories themselves — they only re-expose GetIt singletons and add reactivity
  (`StreamProvider`, `FutureProvider`) on top.
- **Balances are never stored.** `BalanceCalculator` (see §7) derives every balance
  figure at read time from the full transaction history. No `balance` column exists
  anywhere in the schema.
- **Projections (`lib/projections/`) are the read-model layer.** Screens never consume
  raw `TransactionModel`/`PersonModel` lists directly for display logic; feature-level
  aggregators/engines (`DashboardAggregator`, `PeopleViewModel`, `ReportEngine`,
  `TransactionsViewModel`) join repository data into projection objects
  (`PersonSummary`, `TransactionDetails`, `DashboardSummary`, `ReportsOverview`, etc.)
  that implement the empty marker interface `Projection`. Projections are always
  recomputed from repository data, never mutated or persisted.
- **MVVM-ready, not textbook MVVM.** Each feature has a `providers/` folder with a
  `StateNotifier`/`Notifier`-based "ViewModel" (e.g. `PeopleViewModel`,
  `TransactionsViewModel`, `ReportsViewModel`, `DashboardViewModel`,
  `BackupViewModel`) that owns feature state and talks to repositories; screens are
  `ConsumerWidget`/`ConsumerStatefulWidget`s that watch these. This matches the
  README's "MVVM-ready" framing and is now the actual implementation, not just scaffolding.
- **Data flow once UI exists (already true today):** UI widget → `ref.watch`
  a feature ViewModel/provider → ViewModel calls a repository interface method
  (often via `ref.watch(xRepositoryProvider)`) → repository impl issues a Drift query
  against `AppDatabase` → SQLite. Reactive screens (People, Transactions, Dashboard)
  subscribe through `StreamProvider`s backed by `TransactionRepository.watchAll` /
  `watchByPersonId`, so any write from anywhere in the app (including backup/restore)
  propagates to every open screen automatically.
- **Backup/Export/Restore are separate top-level modules** (`lib/backup/`,
  `lib/export/`), not part of the repository layer, but they consume repositories
  (`ExportDataCollectorImpl` takes every repository as a constructor dependency) and
  bypass normal single-row insert/update semantics via `deleteAll()` methods on
  `PeopleRepository`/`CategoryRepository`/`TransactionRepository` reserved for restore.

## 2. Full Project Structure

`lib/` — 19,309 lines across ~140 files (excludes generated `app_database.g.dart`).

### `lib/app/`
- `app_shell.dart` — Root `ConsumerWidget`; hosts Dashboard/People/Reports/Settings
  screens in an `IndexedStack` behind a Material 3 `NavigationBar`, keyed off `appTabProvider`.
- `app_tab_controller.dart` — `AppTab` enum (home/people/reports/settings) +
  `AppTabController extends Notifier<AppTab>` + `appTabProvider`, so any nested widget
  can switch top-level tabs without a reference to `AppShell`.

### `lib/core/`
- `constants/app_constants.dart` — App name, default currency (`INR`/`₹`), DB file
  name, `databaseSchemaVersion = 7`, app version string.
- `constants/default_categories.dart` — `DefaultCategoryDefinition` + the 11 seeded
  categories (Electricity, WiFi, Recharge, Medicine, Travel, Food, Shopping, Cash,
  Bank, UPI, Other) with icon/color.
- `constants/default_people.dart` — `DefaultPersonDefinition` + the 2 seeded people
  (Nani, Sudha), both `PersonType.permanent`.
- `constants/enums.dart` — `PersonType`, `PersonStatus`, `TransactionType`,
  `AppThemeMode`, `BackupFrequency`.
- `database/app_database.dart` — `@DriftDatabase` `AppDatabase` class: schema version,
  migration strategy, first-run seeding, `seedAppInfo()`.
- `database/app_database.g.dart` — Drift-generated table/companion/DB code (4,694
  lines; do not hand-edit; regenerated by `build_runner`).
- `database/converters/enum_converters.dart` — `TypeConverter<T, String>` for each enum
  stored as text (`PersonTypeConverter`, `PersonStatusConverter`,
  `TransactionTypeConverter`, `AppThemeModeConverter`, `BackupFrequencyConverter`),
  each falling back to a safe default on unknown stored values.
- `database/mappers/entity_mappers.dart` — `EntityMappers` abstract final class:
  `toX(entity)` (Entity → domain Model) and `toXCompanion(model)` (Model → Drift
  Companion) for all six tables.
- `database/migrations/database_migrations.dart` — `DatabaseMigrations.onUpgrade`,
  stepping version-by-version (see §3).
- `database/tables/*.dart` — six Drift `Table` definitions (see §3).
- `services/service_locator.dart` — `setupServiceLocator()` / `resetServiceLocator()`,
  the GetIt composition root (see §6).
- `theme/app_theme.dart` — Material 3 `ThemeData` (light/dark), seed color
  `0xFF2E7D32`, `resolveThemeMode(AppThemeMode)`.
- `utils/balance_calculator.dart` — Core balance/running-balance/own-pocket math (§7).
- `utils/balance_colors.dart` — `BalanceColors`: fixed green/red for positive/negative
  balance (not theme-derived), neutral theme color when a person has no transactions.
- `utils/currency_formatter.dart` — `CurrencyFormatter.format(double)`: thousands-grouped
  string with the app's fixed `₹` symbol (no `intl` dependency).
- `utils/friendly_date.dart` — `FriendlyDate.format(DateTime)` → `"11 Jul 2026"` style,
  locale-independent, hand-rolled month table.
- `utils/person_display_order.dart` — `PersonDisplayOrder`: gapped integer ordering
  scheme (`step = 1000`) for future drag-and-drop reordering; `appendAfter`, `between`.
- `utils/relative_date.dart` — `RelativeDate.format(DateTime, {now})` → "Today" /
  "Yesterday" / "N days ago" / falls back to `FriendlyDate` beyond a week.
- `utils/transaction_aggregator.dart` — `TransactionAggregator`: pure grouping/filtering
  helpers (`groupByPerson`, `filterByDateRange`, `runningBalancesById`,
  `ownPocketByTransactionId`, `mostFrequentKey`) shared by People, Dashboard, Export.
- `utils/uuid_generator.dart` — `UuidGenerator.generate()`: dependency-free RFC 4122 v4
  UUID using `Random.secure()`.

### Migration strategy (`DatabaseMigrations.onUpgrade`)
Sequential, one `case` per version, `from+1` → `to`:
- **v2**: `createTable(appInfo)` + `seedAppInfo()`.
- **v3**: `alterTable(TableMigration(transactions))` — rebuilds the table because
  SQLite can't alter an FK action in place; changes `categoryId`'s `onDelete` from
  `SET NULL` to `RESTRICT`. Pre-existing null `categoryId` rows are left as-is.
- **v4**: `addColumn(people, people.avatarSeed)` — plain nullable column add.
- **v5**: `addColumn(people, people.displayOrder)` then backfills every existing
  person's `displayOrder` sequentially (ordered by `id`) using
  `PersonDisplayOrder.initial + index * PersonDisplayOrder.step`.
- **v6**: `createIndex` for all three `transactions` indices (additive only).
- **v7**: `addColumn` × 2 on `settings` — `autoBackupIntervalDays` +
  `autoBackupDirectory`, both nullable (null = off/not-chosen), no backfill, so
  upgraded installations keep automatic backup off until explicitly enabled.
- Unhandled version → `throw StateError`.

`onCreate` runs `m.createAll()` then seeds default categories/people/settings via
`AppDatabase._seedDefaultData()` and `seedAppInfo()`. `beforeOpen` always issues
`PRAGMA foreign_keys = ON`.

- `TransactionModel.personId` → `PersonModel.id` (required; every transaction belongs

### 5.1 Backup & Export Subsystem

`lib/export/` builds a self-contained bundle from repository data; `lib/backup/`
wraps it for the Backup & Restore feature (zips it, records it, and reverses it on
restore). Neither module is itself a "repository" — `ExportDataCollectorImpl` takes
all five repositories as constructor dependencies, and `RestoreServiceImpl` writes
back through `PeopleRepository`/`CategoryRepository`/`TransactionRepository`/
`SettingsRepository`/`AppInfoRepository` plus a direct `AppDatabase.transaction`.

**Bundle format** — a `.zip` named `FamilyLedger_Backup_YYYY-MM-DD_HH-mm.zip`
(`BackupConstants.backupFileName`), flat contents (no wrapper folder):

| File | Format | Content |
|---|---|---|
| `metadata.json` | JSON object | Bundle metadata incl. `exportFormatVersion`, `databaseSchemaVersion`, counts, `checksum`, `includedFiles` manifest. Read first on import. |
| `schema.json` | JSON object | Full field-by-field self-documentation (`ExportSchemaCatalog.document`). |
| `people.json` / `categories.json` / `transactions.json` | JSON array | One object per row, export field names (see mapping below). |
| `settings.json` / `app_info.json` | JSON object | Single-record. |
| `ledger.csv` | CSV, UTF-8 BOM, CRLF, RFC 4180 quoting | Denormalized transaction rows (names resolved). |
| `README.md` | Markdown | Human summary + restore instructions. |
| `attachments/` | raw files | `person_<id>.<ext>` photos, `transaction_<id>.<ext>` attachments. |

`exportFormatVersion` (bundle format) and `databaseSchemaVersion` (Drift schema) are
tracked independently — one can change without the other.

**Field renaming** (export → domain), reversed by `ImportConverters`: e.g. Person
`personIdentifier`→id, `fullName`→name, `contactType`→type, `lifecycleStatus`→status,
`photographFileName`→photoPath (bare name, not device path), `sortPosition`→
displayOrder; Transaction `amount` is **signed** in the export (negative for
`expensePaid`) vs. the domain's always-positive-magnitude-with-type convention —
`TransactionExportMapperImpl` uses `BalanceCalculator.signedAmount`, `ImportConverters
.toTransaction` negates it back for `expensePaid`.

**Checksum**: `ExportChecksum.combinedSha256` joins the *compact* (non-pretty-printed)
JSON of people/categories/transactions/settings/app_info, in that fixed order, with
`\n`, then SHA-256s the UTF-8 bytes → hex digest stored as `metadata.checksum`.
`metadata.json`, `schema.json`, `ledger.csv`, `README.md` are excluded from the hash.

**Import validation** (`ImportValidatorImpl.validate`, sequential, first failure wins,
temp extraction dir always cleaned up):
1. Extract zip → `corruptedArchive` on failure (or `permissionDenied` if a
   `FileSystemException` looks permission-related); includes a **zip-slip guard**
   (`ZipArchiverImpl.extractZip` requires every entry's resolved path stay within the
   extraction root via `p.isWithin`, else `FormatException`).
2. All 6 required files present → else `missingFiles`.
3. Files readable → else `permissionDenied`.
4. Valid JSON → else `corruptedJson`.
5. `metadata.json` parses into `ExportMetadataModel` → else `corruptedJson`.
6. `exportFormatVersion` ∈ `BackupConstants.supportedExportFormatVersions` (`{1}`) →
   else `unsupportedFormatVersion`.
7. Recomputed checksum matches `metadata.checksum` → else `checksumMismatch`.
8. people/categories/transactions are JSON arrays of objects → else `corruptedJson`.
9. Each row converts via `ImportConverters` (catches any exception, not just
   `FormatException`) → else `corruptedJson`.
10. IDs unique within each list → else `duplicateIdentifiers`.
11. Every transaction's `personId`/`categoryId` resolves to a real row → else
    `danglingReference`.
12. Success → `ImportValidationSuccess(ImportBundleModel(...))`.

**Restore** (`RestoreServiceImpl.restore`): on validation failure, throws
`BackupImportException(reason, message)` without touching the DB. On success:
resolves attachment file names into the permanent `<appDocuments>/attachments/`
directory (missing source file → `null`, not a hard failure), then inside one
`AppDatabase.transaction()`: deletes transactions → people → categories (FK-safe
order), re-inserts categories → people → transactions, updates settings, calls
`AppInfoRepository.recordRestore(now)`. All-or-nothing.

**Backup** (`BackupServiceImpl.createBackup`): probes destination directory
writability first (write+delete a probe file, fails fast with `BackupExportException`
otherwise), exports to a temp staging dir via `ExportService.exportAll`, zips it,
records the result via `BackupRepository.insert` + `AppInfoRepository.recordBackup`.

**Stale doc comments**: `AppInfoExportMapper`, `CategoryExportMapper`,
`SettingsExportMapper` interface files claim "no implementation exists yet" — false;
`AppInfoExportMapperImpl`/`CategoryExportMapperImpl`/`SettingsExportMapperImpl` are
fully implemented and wired into `ExportDataCollectorImpl`. Do not trust these
comments; trust the `impl/` files.

## 7. Balance Calculation Logic

Formula, exactly as coded in `lib/core/utils/balance_calculator.dart`:

```
Balance = Advance Received − Expense Paid + Money Returned ± Adjustments
```

Implemented as a per-transaction signed-amount fold, not a single formula call:

```dart
static double _signedAmount(TransactionModel transaction) {
  final amount = transaction.amount;
  return switch (transaction.transactionType) {
    TransactionType.advanceReceived => amount,
    TransactionType.expensePaid => -amount,
    TransactionType.moneyReturned => amount,
    TransactionType.adjustment => amount,   // signed; caller can store negative amounts
  };
}
```

Public API (`abstract final class BalanceCalculator`):
- `calculateBalance(List<TransactionModel>)` — sums `_signedAmount` over the whole list.
  Order-independent (a plain sum), used by `TransactionRepository.calculateBalance`.
- `signedAmount(TransactionModel)` — public wrapper for a single transaction's
  contribution (used by report engines to accumulate section totals).
- `runningBalances(List<TransactionModel> chronologicalTransactions)` — **requires
  oldest-first, single-person input**; returns a same-length list where `result[i]` is
  the balance immediately after transaction `i`. Never stored — recomputed every read.
- `ownPocketPortions(List<TransactionModel> chronologicalTransactions)` — same ordering
  contract as `runningBalances`. For each `expensePaid` transaction, computes how much
  of it exceeded the advance balance available immediately before it (`available =
  balance > 0 ? balance : 0`; `uncovered = amount - available`; clamped to `>= 0`).
  Every non-`expensePaid` transaction contributes `0`. This is what lets the app answer
  "how much of my own money did I actually spend" separately from a plain expense total.
- `calculateBalancesByPerson(List<TransactionModel>)` — one-pass fold into a
  `Map<int personId, double balance>` for a flat, multi-person transaction list.

**Edge cases handled:**
- Order sensitivity: `runningBalances`/`ownPocketPortions` are documented as requiring
  chronological (oldest-first) input; callers reverse newest-first repository results
  before calling (see `TransactionAggregator.runningBalancesById`, which also confirms
  each person's sub-list is reversed independently so one person's history never
  pollutes another's running balance).
- Own-pocket clamping: an expense fully covered by existing advance contributes exactly
  `0`; an expense exceeding available advance (including already-negative balance,
  where `available` clamps to `0`) contributes the full uncovered amount.
- Adjustment sign: `adjustment` transactions store a signed amount and pass through
  unchanged — the type itself doesn't force a sign, callers decide.
- Multi-person aggregation: `calculateBalancesByPerson` uses `Map.update(...,
  ifAbsent: ...)` so it works correctly on a completely flat, unsorted, multi-person list.

**Where the logic lives:** exclusively in `BalanceCalculator`. No screen, ViewModel, or
repository re-implements the signed-amount switch — `TransactionRepository
.calculateBalance`, `TransactionAggregator.runningBalancesById`/
`ownPocketByTransactionId`, and every report/dashboard aggregator delegate to it.

**Unit tests:** `test/core/utils/balance_calculator_test.dart` (153 lines, 5 tests) and
`test/core/utils/own_pocket_test.dart` (212 lines, 7 tests) — see §8 for the full
scenario list.

## 10. What's NOT Built Yet

Correcting the README's outdated "Phase 1 — no UI screens implemented yet" framing:
**Dashboard, People, Reports, Transactions, and Backup are all fully implemented,**
wired end-to-end to real repositories, with no TODOs or placeholder branches found
in any of their ~90 files. The genuine gaps are narrower and more specific:

- **Settings screen is a real stub.** `SettingsScreen` has exactly one working
  entry — a `ListTile` opening Backup & Restore — followed by explicit placeholder
  text: *"More settings coming in future phases."* Theme selection, currency
  selection, and automatic-backup-frequency configuration have no UI at all, even
  though their data model (`SettingsModel.theme`/`currency`/`backupFrequency`) and
  repository (`SettingsRepository`) are fully built and tested.
- **Currency is hardcoded**, not wired to `SettingsModel.currency` — `CurrencyFormatter`
  uses `AppConstants.defaultCurrencySymbol` (`₹`) unconditionally everywhere it's
  called. Its own doc comment calls this "a natural follow-up once the Settings
  screen exists" — the screen now exists but doesn't yet expose a currency picker,
  so this remains open.
- **No category management UI.** No screen lets a user add/edit/delete a category —
  only the 11 seeded defaults exist. `categoriesListProvider` is deliberately a
  one-shot `FutureProvider` rather than a reactive stream *because* of this; its own
  doc comment says to upgrade it to a `StreamProvider` if a future screen starts
  mutating categories. `CategoryRepository.delete`'s reassignment logic is fully
  built and tested but has no caller.
- **No theme picker UI** despite `AppTheme`/`AppThemeMode` being fully implemented —
  `main.dart` hardcodes `themeMode: ThemeMode.system`, never reading
  `SettingsModel.theme`.
- **Automatic backup is check-on-open only** (no background/WorkManager execution):
  `AutoBackupService` (`lib/backup/services/`) runs from `AppShell` at launch/resume,
  backing up to `SettingsModel.autoBackupDirectory` when the last backup is ≥
  `autoBackupIntervalDays` days old, then rotating so only the newest

  so the export bundle format (settings.json) stays unchanged.
- **Drag-and-drop person reordering.** `PersonDisplayOrder.between()` is implemented

## 11. Conventions to Follow

- **Model classes are hand-written, not code-generated.** Every `lib/models/*.dart`
  class manually implements `copyWith`, `toJson`/`fromJson`, `==`/`hashCode` — no
  `freezed`/`json_serializable`. New models should follow the same explicit pattern.
- **`abstract final class`** is used for static-only utility/constant holders
  (`BalanceCalculator`, `TransactionAggregator`, `AppConstants`, `EntityMappers`,
  `CurrencyFormatter`, `FriendlyDate`, `RelativeDate`, `PersonDisplayOrder`,
  `UuidGenerator`, `AppTheme`, `DatabaseMigrations`) — never instantiated, never
  subclassed.
- **`abstract interface class`** is used for every repository contract and for the
  `Projection` marker — pure interfaces, no default method bodies.
- **Repository interfaces live one level above `impl/`**; naming is always
  `XRepository` / `XRepositoryImpl`, one file each, `impl/` subfolder. Same pattern
  in `lib/backup/services/` (`XService` / `impl/XServiceImpl`) and
  `lib/export/services/` and `lib/export/converters/`.
- **Nullable `id` convention**: every persisted model's `id` is `int?`, null before
  insert. Repository `insert()` methods return the generated `int` id; `update()`
  methods return `bool` (rows-affected > 0).
- **`deleteAll()` methods exist only for restore.** They bypass normal delete
  validation (e.g. `CategoryRepository.deleteAll()` skips the reassignment
  `delete()` normally requires) and are documented as restore-only — do not call
  them from ordinary feature code.
- **Balance/derived figures are never persisted.** Any new "figure" (a sum, a
  count, a most-recent-X) should be computed from repository data at read time,
  the same way every existing projection does — do not add a stored/cached column
  for something derivable from `transactions`.
- **Reactive vs. one-shot providers**: use `StreamProvider` backed by a repository
  `watchX()` when multiple features/screens need to stay in sync with writes from
  anywhere (see `transactionsStreamProvider`); use `FutureProvider` only when the
  underlying data is not mutated by any screen yet (see `categoriesListProvider`'s
  own doc comment on when to upgrade it).
- **`autoDispose` + `.family`** for reactive providers scoped to an argument that
  can have many transient instances (e.g. `personTransactionsStreamProvider`) —
  contrast with app-wide singleton streams like `transactionsStreamProvider`, which
  deliberately omit `autoDispose` since always-alive tabs share it.
- **Doc comments explain *why*, not *what***, especially for non-obvious schema/
  ordering/migration decisions (see `people_table.dart`'s `displayOrder` comment,
  `transactions_table.dart`'s index-rationale comment, every migration `case` in
  `database_migrations.dart`). Follow this style — a future migration or schema
  change should explain the constraint it's working around, not just restate the
  Drift API call.
- **Trailing commas and const-first** are enforced by `analysis_options.yaml`
  (`require_trailing_commas`, `prefer_const_constructors`, `prefer_const_declarations`).
- **Testing**: in-memory `AppDatabase.forTesting(NativeDatabase.memory())` (or
  equivalent) for database-touching tests; see `test/helpers/test_database.dart` and
  `test/helpers/test_repositories.dart` for the shared setup. Full test conventions
  in §8.

### Feature-package conventions (from the UI layer)
- **Every feature splits transient UI state from persisted-data state into two
  providers**: a `*QueryController`/`*FilterController` (`Notifier`, holds
  search/sort/filter selections only) and a `*ViewModel` (`AsyncNotifier`, holds the
  actual data from repositories). A third `Provider` (e.g. `filteredPeopleProvider`,
  `filteredTransactionsProvider`) combines the two via a pure `*QueryEngine`/
  `*FilterEngine.apply(...)` static method. Follow this three-part split for any new
  filterable/sortable list screen rather than folding filter state into the view model.
- **Query/filter/report engines are pure static classes** (`PeopleQueryEngine`,
  `TransactionQueryEngine`, `ReportFilterEngine`, `ReportEngine`) — no Riverpod, no
  I/O, take already-loaded data in and return already-filtered/sorted data out. This
  is what makes them unit-testable without a database or widget harness (see §8 — these
  are consistently among the most thoroughly tested files).
- **Domain-specific exceptions per feature**, not generic errors: `EmptyPersonNameException`,
  `DuplicatePersonNameException(name)`, `PersonHasTransactionsException`,
  `InvalidTransactionAmountException`, `RemarkTooLongException({maxLength})`,
  `BackupExportException(message)`, `BackupImportException(reason, message)`. Each
  implements `Exception`, carries a `message` getter, and is caught by name in the
  calling widget to show an inline/snackbar error — never a generic `catch (e)`
  string. Follow this pattern for any new validated mutation.
- **`copyWith` with an explicit-null escape hatch** where a field legitimately needs
  to be cleared: `ReportFilter.copyWith` uses a private `_unset` sentinel object so
  `null` can mean "set to null" instead of "leave unchanged"; `TransactionQuery.copyWith`
  instead uses explicit `clearX` bool flags for the same purpose. Either is
  acceptable — pick one per new state class, don't invent a third pattern.
- **Static `show(...)` factory on dialogs/sheets** that own their own presentation:
  `AddEditPersonDialog.show(context, {initialPerson})`,
  `AddEditTransactionSheet.show(context, {personId, initialTransaction})`. Callers
  never call `showDialog`/`showModalBottomSheet` directly for these.
- **Search inputs are debounced** (`Timer`, 300ms in `ReportFilterBar`) before
  writing to the query controller; immediate clear bypasses the debounce. Follow
  this for any new free-text filter to avoid rebuilding the list on every keystroke.
- **Reused-not-recomputed balances**: any new feature needing a person's balance
  should read it from `PersonSummary` (via `peopleViewModelProvider`) rather than
  re-deriving it from raw transactions — `DashboardAggregator` and `ReportEngine`
  both do this specifically to avoid two slightly-different balance implementations
  drifting apart.
- **Private widget classes are prefixed with `_`** and typically live in the same
  file as the public widget that uses them (e.g. `_PeopleList`, `_ErrorState` in
  `people_screen.dart`) rather than being split into their own files, unless reused
  across multiple screens (in which case they move to `widgets/` and drop the underscore).
